## 1. 背景

当前前端已经具备完整功能，但 UI 层存在三个结构性问题：

1. `src/App.svelte` 过大，页面装配、业务状态、动效、面板 UI 混在一起。
2. `src/app.css` 过大，样式职责分散，重复定义较多，难以稳定迭代。
3. 现有 UI 以手写样式为主，缺少统一的设计系统、组件原语和可复用的变体体系。

这会直接带来几个后果：

1. 小改动也容易牵动多个区域。
2. 新增 UI 能力时，很难保持一致的层级、间距、反馈和可访问性。
3. 面板、按钮、输入、开关、列表项等通用模式没有统一抽象，长期会继续复制样式和逻辑。

这次改造的目标，不是单纯“换成 Tailwind”，也不是把现有 CSS 机械翻译成 class，而是把当前前端升级成一套可持续维护的 UI 系统。

---

## 2. 改造目标

### 2.1 总目标

基于 `shadcn-svelte + Tailwind CSS + Bits UI` 重建前端 UI 层，形成“设计令牌 + 组件原语 + 业务复合组件 + 分域状态”的稳定结构。

### 2.2 具体目标

1. 保留当前业务能力和后端接口，不改变下载、播放、歌词、通知等功能语义。
2. 用 `Tailwind CSS` 取代大部分页面级和组件级手写样式。
3. 用 `shadcn-svelte` 提供基础组件原语和可访问性能力。
4. 用项目自己的设计令牌和变体体系，避免直接套用默认 shadcn 风格。
5. 按 `library / player / download / shell / env` 领域拆分前端结构。
6. 将 `App.svelte` 收敛成装配层。
7. 清理阻塞式 `alert()`、重复样式、重复交互逻辑和大块内联模板。

---

## 3. 非目标

这次改造不包含以下事项：

1. 不迁移到 `SvelteKit`。
2. 不重写 Rust / Tauri 后端。
3. 不修改现有下载任务模型和播放状态契约。
4. 不在第一阶段重做信息架构和核心交互流程。
5. 不追求完全去除所有自定义样式。

---

## 4. 技术选型结论

### 4.1 结论

UI 改造采用下面这套组合：

1. `Tailwind CSS` 作为主样式层。
2. `shadcn-svelte` 作为基础组件源码分发层。
3. `Bits UI` 作为底层无样式交互原语来源。
4. `@humanspeak/svelte-motion` 继续承担动效实现。

### 4.2 选型原因

### `awesome-design-systems`

该仓库适合作为设计系统参考索引，而不是直接复用的模板仓库。它的价值在于帮助我们抽取规则：

1. 如何建立统一的视觉语言。
2. 如何定义组件层级和使用边界。
3. 如何把“设计系统”和“UI 组件库”区分开。

因此，这次改造不是照搬某个现成设计系统，而是参考其组织方式，建立项目自己的 UI 系统。

### `shadcn-svelte`

`shadcn-svelte` 适合这个项目，原因是：

1. 它不是黑盒组件库，更适合桌面应用这种需要深度定制视觉的项目。
2. 它基于 `Bits UI`，无障碍和交互基础较完整。
3. 它能快速补齐 `Sheet / Select / Switch / Progress / Tooltip / Toast` 这类通用能力。
4. 它允许保留项目自己的视觉语言，不会强迫页面长成统一模板风格。

### `Tailwind CSS`

`Tailwind CSS` 适合承担这次改造的主样式层，原因是：

1. 当前样式重复较多，Tailwind 更适合把 spacing、radius、border、state 收敛成稳定规则。
2. 业务组件拆分后，Tailwind 能让局部样式跟组件一起收口，不再依赖超长全局 CSS。
3. 可以与项目级 CSS 变量结合，继续支持动态主题色和亮暗色模式。

### 4.3 从 `awesome-design-systems` 抽取的方法论

基于该仓库的定义和标签体系，这次改造要明确一件事：

设计系统不等于组件库。

一个成熟设计系统，至少包含下面几层：

1. 原则和最佳实践。
2. foundations，例如 color、spacing、typography、elevation、radius。
3. components。
4. patterns。
5. content / voice & tone。
6. designers kit 或设计资产。
7. source code。
8. 发布、弃用、贡献和治理机制。

这意味着本项目不能只做两件事：

1. 安装 `shadcn-svelte`。
2. 把旧 CSS 改成 Tailwind。

还必须补齐这些缺口：

1. 设计令牌。
2. 模式层。
3. 内容规范。
4. 组件状态管理。
5. 文档和治理。

### 4.4 从代表性设计系统得到的直接启发

结合 `awesome-design-systems` 中收录的成熟系统，可以提炼出几条和本项目直接相关的规则：

1. `Atlassian Design System` 把系统拆成 `foundations / components / patterns / tools / content`，说明 UI 系统必须覆盖设计、开发、内容三个角色。
2. `Cloudscape` 强调 visual context，说明同一组件在不同表面和层级里不能只靠一套固定颜色硬撑。
3. `Spectrum 2` 强调 contextual、cohesive、performant、collective，说明设计系统不是一次性定稿，而是围绕上下文、性能和协作持续演进。

## 6. 改造原则

### 6.1 设计系统先行

先建立设计令牌、组件原语和变体规则，再迁移具体页面。

### 6.2 结构拆分优先于样式替换

先解决 `App.svelte` 的职责堆积，再做分块 UI 重构。否则 Tailwind class 只会堆进旧结构里。

### 6.3 通用件标准化，核心件定制化

以下通用件优先复用 `shadcn-svelte`：

1. Button
2. Badge
3. Sheet
4. Select
5. Switch
6. Scroll Area
7. Progress
8. Tooltip
9. Dialog / Alert Dialog
10. Sonner / Toast
11. Skeleton
12. Tabs
13. Slider

以下核心件保持项目定制：

1. AlbumCard
2. SongRow
3. PlayerDock
4. AlbumStage
5. TopToolbar 的模式切换段控件

### 6.4 保留动态主题色

当前“根据专辑封面提取主题色”的能力必须保留。Tailwind 负责结构和常规样式，颜色仍通过 CSS 变量驱动。

### 6.5 改造期间保持功能稳定

所有阶段都以“行为不变”为前提推进。视觉可以升级，交互语义不能漂移。

### 6.6 视觉上下文优先

不能把所有组件都当成“放在白底页面上的通用组件”。

本项目至少存在这些视觉上下文：

1. 专辑侧栏
2. 主内容工作区
3. 设置 / 下载右侧面板
4. 播放器 Dock
5. 歌词 / 队列浮层
6. 状态类表面，例如下载失败、下载进行中、选中态

设计令牌和组件变体要能适配这些上下文，而不是写死一套背景和边框。

### 6.7 内容和反馈属于系统本身

按钮文案、空状态、错误信息、成功反馈、时间和数量表述，都是设计系统的一部分。

这次改造要把下面这些内容纳入系统治理：

1. 操作按钮命名
2. 错误提示格式
3. 成功反馈格式
4. 空状态文案
5. 时间 / 数量 / 状态文本
6. toast 文案规范

### 6.8 Apple 化风格边界

基于当前已确定的方向，这次 UI 风格不是纯粹复刻 Apple Music，也不是普通 macOS 工具界面，而是：

`macOS 应用骨架 + Apple Music 的内容表达`

风格边界已经固定为：

1. 整体路线：`精致折中`
2. 材质强度：`适中`
3. 主题色策略：`保留动态主题色，但统一降饱和`
4. 排版策略：`明显 Apple 化`

将这些选择翻译成设计约束后，得到下面这套规则：

1. 明显的毛玻璃和半透明材质只放在 `右侧面板`、`下载任务面板`、`底部 Dock`、`播放器浮层`。
2. 主工作区保持更干净的实色或弱渐层表面，不做整页玻璃化。
3. 专辑主题色继续由封面提取，但只用于氛围、重点状态和局部高亮，不允许把整个页面染成高饱和颜色。
4. 大标题、页面标题、面板标题、播放器当前曲目区域明显更像 Apple 的排版层级；列表和工具区域仍保留效率。
5. 阴影、边框、高光都要更轻，避免看起来像“厚重的拟物卡片”或“通用 web SaaS 模板”。
6. 交互动画要像系统过渡，不做弹跳和夸张放缩。

### 6.9 明确避免的视觉误区

为了保证“更接近 Apple”而不跑偏，这次改造明确避免下面这些方向：

1. 不做全屏毛玻璃。
2. 不做高饱和、大面积专辑色铺底。
3. 不做厚重阴影和粗边框。
4. 不做 iOS 风格的大圆角大留白，导致桌面端信息密度失衡。
5. 不做 marketing page 式大段展示排版，削弱工具属性。
6. 不直接照搬 shadcn 默认主题外观。

## 8. 设计系统层级

这次改造后，UI 层应按多层组织：

### 8.1 设计令牌层

负责颜色、圆角、阴影、间距、动画、层级。

来源：

1. 当前 `app.css` 中已有的 CSS 变量。
2. 当前播放器和专辑舞台中反复出现的渐变、阴影、透明度规则。

建议沉淀的令牌类别：

1. `surface`
2. `text`
3. `accent`
4. `border`
5. `shadow`
6. `radius`
7. `motion`
8. `density`

在“精致折中”的方向下，令牌要进一步细分为下面这些语义表面：

1. `surface.window`
   整个窗口的基础表面。
2. `surface.sidebar`
   左侧专辑区，更接近 macOS 侧栏。
3. `surface.workspace`
   主工作区，保持干净。
4. `surface.sheet`
   设置和下载右侧面板，材质感更强。
5. `surface.dock`
   底部播放器 Dock，材质感更强。
6. `surface.flyout`
   歌词和播放列表浮层，比 Sheet 更轻。
7. `surface.state`
   下载中、失败、选中等状态表面。

在颜色和材质上，还要补充下面这些约束：

1. 动态专辑色统一做 `降饱和`、`提亮`、`压对比` 处理。
2. 中性色要向暖灰或冷灰轻微偏移，避免纯黑纯白。
3. 玻璃材质依赖 `透明度 + 高光 + 弱边框 + 轻阴影` 组合，不依赖重模糊本身。
4. 阴影要更像浮层分层，不像悬浮卡片堆叠。

### 8.2 原语组件层

基于 `shadcn-svelte / Bits UI` 提供基础结构和可访问性能力。

### 8.3 项目变体层

负责把“同一个 Button 在不同区域长什么样”统一下来。

例如：

1. toolbar icon button
2. primary action button
3. subtle action button
4. player control button
5. destructive mini action

在 Apple 化方向下，项目变体还要补这几类：

1. segmented mode control
2. macOS-style sidebar item
3. sheet section header
4. dock utility button
5. list row action button
6. tonal status badge

### 8.4 业务复合组件层

面向页面直接使用的组件：

1. `AlbumCard`
2. `SongRow`
3. `PlayerDock`
4. `SettingsSheet`
5. `DownloadTasksSheet`

### 8.5 模式层

模式层位于“单个组件”和“完整页面”之间。

它负责解决这些问题：

1. 下载任务列表应该怎么组织。
2. 设置面板中的分组、说明、危险操作怎么排布。
3. 歌词面板和播放列表面板的共同结构是什么。
4. 多选下载工具条应该如何出现、消失、禁用和反馈。

本项目至少应沉淀这些模式：

1. 侧栏列表模式
2. 主工作区模式
3. 右侧 Sheet 模式
4. 状态反馈模式
5. 播放器浮层模式
6. 空状态 / 加载状态 / 错误状态模式

### 8.6 内容设计层

内容设计层负责统一界面文案，不让提示语和状态语句在不同区域各说各话。

至少需要覆盖：

1. 按钮动词
2. 面板标题
3. 错误信息
4. 成功反馈
5. 重试 / 取消 / 清理 / 删除等风险动作
6. 空状态
7. 加载状态

Apple 化方向下，内容设计还要遵守下面这些语气规则：

1. 句子更短。
2. 少解释，少感叹。
3. 面板标题更像系统功能名，而不是营销语。
4. 空状态和错误状态避免过度安抚或技术细节堆砌。
5. 风险动作文案明确直接，不绕。

### 8.7 治理与工具层

治理与工具层负责保证系统不是“写完一版组件就散掉”。

至少需要：

1. 组件 owner
2. 组件状态标记
3. 设计令牌文档
4. 组件使用规范
5. 内容规范
6. lint / check / review 规则
7. 迁移和弃用策略

## 9. 样式迁移策略

### 9.1 保留项

以下样式能力短期内继续保留在全局 CSS 中：

1. 亮色 / 暗色主题根变量。
2. 专辑主题色变量注入。
3. Tauri drag region。
4. 覆盖第三方滚动条样式。
5. 少量无法通过 class 简洁表达的复杂视觉特例。

### 9.2 迁移项

以下内容优先迁移到 Tailwind class + variants：

1. 布局容器
2. 面板结构
3. 表单样式
4. 通用按钮
5. 状态标签
6. 列表项 hover / active / selected / disabled
7. 面板内的标题、说明、副文案

### 9.3 最终目标

`src/app.css` 最终只保留：

1. Tailwind 入口。
2. 设计令牌变量。
3. 全局 reset。
4. Tauri / 浏览器兼容性特例。
5. 少量全局覆盖。

### 9.4 Apple 化样式落地规则

样式迁移时，要把“Apple 风格”翻成可以执行的规则，而不是停留在感觉层面。

具体规则如下：

1. 页面标题和面板标题使用更大字号，但控制字重，不靠纯粗体拉层级。
2. 列表项、工具条、任务项统一减少无意义背景块，更多使用 hover/selected 的轻状态变化。
3. 大部分组件默认边框更轻，圆角更细腻，不使用过大的卡片圆角。
4. 右侧面板和底部 Dock 使用一致的玻璃材质公式，避免每个地方一套参数。
5. 主工作区背景保持平整，允许有弱渐层和弱主题色雾化，但不出现明显玻璃面。
6. 专辑舞台只保留一层主题氛围，不做复杂多层发光。
7. 交互动效统一使用更短、更稳的 easing，不使用 bounce。

## 10. 组件映射方案

## 10.0 区域级视觉改造总则

在当前风格方向下，各区域统一遵守下面这套分工：

1. `左侧专辑区`
   更像 macOS 侧栏。安静、中性、轻选中态。
2. `主内容工作区`
   更干净，主要承载内容和阅读，不承担强材质表现。
3. `右侧设置 / 下载面板`
   材质感最明显，负责把 Apple 式分层带进界面。
4. `底部播放器 Dock`
   是整站第二个最有 Apple Music 气质的地方。
5. `歌词 / 队列浮层`
   视觉上继承 Dock，但更轻，避免抢占主视觉。
6. `状态表面`
   统一弱化色彩，不做强烈警报式大红大绿块。

### 10.1 顶部工具栏

现状：

1. 模式切换
2. 刷新
3. 下载任务入口
4. 设置入口

改造后：

1. 图标按钮统一基于 `Button` 变体。
2. badge 改用 `Badge`。
3. tooltip 统一接入。
4. 模式切换保留自定义 segmented control。
5. 工具栏本身从“悬浮 web 工具条”改成“更像 macOS 控件条”的轻材质条。

视觉安排：

1. 工具栏保留悬浮感，但阴影减轻。
2. 模式切换像 Apple segmented control，强调当前模式，不强调描边。
3. 图标按钮更轻，默认接近透明，只在 hover / active 时出现轻底色。
4. badge 更小、更克制，像系统计数，不像通知角标。

### 10.2 设置面板

现状：

1. 手写 overlay
2. 手写 panel
3. 手写 select / switch / button

改造后：

1. 容器改为 `Sheet`
2. 输出格式改为 `Select`
3. 开关改为 `Switch`
4. 按钮改为 `Button`
5. 提示文字、分组、分隔规则统一
6. 视觉上作为全站材质感最明显的区域之一。

视觉安排：

1. 面板表面采用明显但不过量的玻璃材质。
2. 分组标题更像系统设置标题，不做营销式副标题。
3. 开关、选择器、按钮都向系统偏好面板靠拢，强调轻和稳。
4. 说明文字更短，避免一组设置下挂太长解释。
5. 危险操作与普通操作要拉开，但不能靠高饱和红色大面积区分。

### 10.3 下载任务面板

改造后：

1. 容器改为 `Sheet`
2. 任务列表区域优先用 `Scroll Area`
3. 进度条改为 `Progress`
4. 状态标签改为 `Badge`
5. 重试 / 取消按钮统一变体
6. 视觉上和设置面板同属一套材质体系，但信息密度更高。

视觉安排：

1. 面板和设置面板共享玻璃材质参数。
2. 任务卡片弱化“卡片感”，更像列表中的分组条目。
3. 进度条更细，状态色更克制。
4. 错误信息不做大红区块，使用轻红字色 + 细状态标记。
5. 历史任务弱化对比，让运行中任务自然成为视觉焦点。

### 10.4 专辑侧栏

改造后：

1. 侧栏容器用 Tailwind 管理布局
2. `AlbumCard` 保持定制组件
3. 聚焦态、选中态、hover 态统一通过 variants 驱动
4. 整体更接近 macOS 侧栏，而不是普通 web 列表栏。

视觉安排：

1. 侧栏表面保持中性、干净，不参与大面积主题染色。
2. 选中态像 Apple 侧栏选中条，使用轻底色和轻文本强化。
3. hover 态只做很轻的提亮，不做明显浮起。
4. 专辑封面缩略图更整洁，圆角更接近系统图标或媒体缩略图比例。
5. 标题与副标题层级更克制，避免同时都很抢。

### 10.5 专辑主区

改造后：

1. hero 区保留定制布局
2. skeleton 改用 `Skeleton`
3. 操作按钮统一为 `Button`
4. 元信息标签统一为 `Badge`
5. 主工作区整体保持干净，是 Apple 化改造里最需要克制的区域。

视觉安排：

1. 专辑舞台保留，但从“强展示舞台”收敛为“内容前景 + 弱主题氛围”。
2. 封面仍是主角，但背景只保留降饱和主题雾感，不做强烈大面积染色。
3. 标题更大、更像 Apple 的内容标题，但说明文本更短、更干净。
4. 元数据和操作区要排得更稳，不堆过多小标签。
5. 内容滚动区背景趋向实色和弱渐层，减少视觉噪音。

### 10.6 曲目列表

改造后：

1. `SongRow` 仍为定制组件
2. 行内下载、选中、播放状态改为统一状态变体
3. 行内小按钮统一从项目按钮变体复用
4. 列表需要兼顾 Apple 排版感和桌面工具效率。

视觉安排：

1. 行高可以略微增加，但不能大幅稀疏，避免损伤下载工具效率。
2. 默认状态更干净，尽量减少每行都带明显底色。
3. 当前播放、选中、多选态统一走轻量背景 + 轻色强调。
4. 行内操作按钮平时更隐，hover 或焦点时再更清晰。
5. 艺术家、副信息和序号的层级重新整理，减少视觉碎片。

### 10.7 播放器 Dock

改造后：

1. 按钮基于 `Button`
2. 进度条基于 `Slider`
3. 歌词 / 播放列表浮层基于 `Popover` 或定制浮层 + `Scroll Area`
4. 下载状态与提示统一接设计令牌和 badge 体系
5. 这是最明显借 Apple Music 的区域之一。

视觉安排：

1. Dock 保留明显材质感，但避免过厚玻璃和重阴影。
2. 当前曲目信息区更像 Apple Music 的紧凑媒体区：封面更精致，文字更稳。
3. 播放控制更集中，次级操作更退后。
4. 进度条更细，thumb 更轻，默认不抢眼。
5. 下载、歌词、播放列表这些附加按钮应更像系统 utility controls。

### 10.8 歌词与播放列表浮层

改造后：

1. 浮层在视觉上继承 Dock。
2. 浮层材质比右侧 Sheet 更轻。
3. 内部列表结构统一，减少两个浮层各写一套样式。

视觉安排：

1. 浮层背景和 Dock 同宗，但对比更低。
2. 当前歌词行和当前播放项的高亮更克制，不用大面积色块。
3. 浮层标题、计数、空状态都走统一的 Apple 化轻排版。
4. 列表内部滚动区更干净，不做过度装饰。

### 10.9 空状态、加载状态、错误状态

改造后：

1. 用统一模式处理所有空 / 加载 / 错误状态。
2. 骨架屏、toast、面板内反馈都进入同一语气和视觉系统。

视觉安排：

1. 空状态更安静，图形和文案都简短。
2. 加载状态更像系统内容占位，不做太多脉冲和闪烁。
3. 错误状态强调可恢复，不做惊吓式视觉。
4. toast 更短、更轻、更像系统反馈，而不是网页通知条。

### 10.10 迁移判定矩阵

不是所有 UI 都应该直接换成现成组件。

可以按下面三类判定：

#### A 类：直接使用 `shadcn-svelte`

适用条件：

1. 通用交互
2. 语义清晰
3. 视觉差异不大
4. 无复杂业务耦合

代表对象：

1. Button
2. Select
3. Switch
4. Sheet
5. Progress
6. Tooltip
7. Dialog
8. Skeleton

#### B 类：基于 `shadcn-svelte` 包一层项目组件

适用条件：

1. 底层交互通用
2. 视觉状态较多
3. 项目里重复出现

代表对象：

1. ToolbarIconButton
2. AppBadge
3. PanelSection
4. StatusToast
5. PlayerControlButton

#### C 类：保留定制组件，只消费令牌和原语

适用条件：

1. 业务耦合强
2. 状态密度高
3. 布局和动画独特
4. 需要动态主题色或复杂上下文适配

代表对象：

1. AlbumStage
2. AlbumCard
3. SongRow
4. PlayerDock
5. LyricsFlyout
6. PlaylistFlyout

## 11. 分阶段实施计划

在执行顺序上，要把“设计系统补全”放进改造本身，而不是只改组件。

### Phase 0：文档和边界冻结

### 目标

把“为什么改、改什么、不改什么”写死。

### 输出

1. 本文档。
2. UI 组件映射表。
3. 设计令牌初稿。
4. 组件分类清单。
5. 组件状态定义。
6. 内容规范初稿。

---

### Phase 1：接入 Tailwind 和 shadcn-svelte

### 目标

建立新 UI 栈的基础设施。

### 主要工作

1. 安装 `tailwindcss` 与 Vite 插件。
2. 配置 `shadcn-svelte`。
3. 修正 `tsconfig.json` 路径映射。
4. 新建 `utils.ts` 与 `cn()`。
5. 初始化基础 UI 原语。
6. 初始化设计令牌文档。
7. 初始化组件指南文档。

### 完成标准

1. 能在当前项目中直接渲染 shadcn 组件。
2. Tailwind 生效。
3. 构建链路通过。
4. 基础原语目录和文档目录已建立。

---

### Phase 2：建立设计令牌和项目变体

### 目标

让项目拥有自己的视觉系统，而不是直接套默认皮肤。

2. 建立 `tokens.ts`。
3. 建立 `variants.ts`。

5. 建立内容设计规则。
6. 建立视觉上下文规则。

### 完成标准

1. 新组件不再直接写散落颜色值。
2. 通用视觉规则开始统一。
3. 同一组件在不同表面下有明确上下文策略。
4. 文案、状态、反馈语句开始统一。

### Phase 3：结构拆分

按 [APP_SVELTE_SPLIT_PLAN_B.md](./APP_SVELTE_SPLIT_PLAN_B.md) 拆分 `App.svelte`。

5. 为 UI 改造建立清晰边界，避免 Tailwind class 继续堆入旧结构。

### 完成标准

### 完成标准

## 12. 风险与控制

### 12.1 主要风险

1. UI 改造和结构拆分同时进行，容易扩大回归面。
2. 播放器和滚动舞台定制程度高，直接套通用组件容易退化体验。
3. 动态主题色如果处理不好，Tailwind 和 CSS 变量会打架。
4. 旧逻辑还在 `App.svelte` 时，Tailwind class 容易继续堆积。

### 12.2 控制策略

1. 每个阶段独立提交，避免超大 PR。
2. 先改标准化区域，再改播放器和舞台。
3. 主题色继续走 CSS 变量，不强行静态化。
4. 所有阶段都以“功能冒烟通过”为合并前提。

### 12.3 组件发布与弃用机制

参考成熟设计系统的治理方式，项目内组件也应标记生命周期。

建议使用四种状态：

1. `draft`
   仅在局部试验，不允许全局复用。
2. `beta`
   可以在新页面中使用，但 API 和视觉仍可能调整。
3. `stable`
   可以作为项目默认方案复用。
4. `deprecated`
   不再推荐使用，只保留迁移窗口。

进入 `stable` 的最低要求：

1. 已接入设计令牌。
2. 已定义变体边界。
3. 已有可访问性检查。
4. 已写入组件说明。
5. 已至少被两个场景复用。

3. `svelte-check` 无阻塞问题。

3. 专辑、曲目、播放器、下载任务的视觉风格统一。

5. 不同视觉上下文中的组件都能保持可读性和一致性。

1. 下载功能全链路正常。
2. 播放器全链路正常。

1. 所有新增可复用组件都能归入 `原语 / 项目变体 / 业务复合组件 / 模式` 之一。

3. 新增 UI 文案遵守统一内容规范。

## 14. 建议的文档协同关系

建议将文档拆成多个职责明确的文件：

1. [FRONTEND_ISSUES.md](./FRONTEND_ISSUES.md)
   负责记录现状问题和优先级。
2. [APP_SVELTE_SPLIT_PLAN_B.md](./APP_SVELTE_SPLIT_PLAN_B.md)
   负责前端结构拆分方案。
3. `UI_REFACTOR_PLAN.md`
   负责 UI 栈、设计系统、组件映射和改造路线。
4. `DESIGN_SYSTEM_TOKENS.md`
   负责令牌定义和语义边界。
5. `COMPONENT_GUIDELINES.md`
   负责组件分类、使用约束和生命周期。
6. `CONTENT_GUIDELINES.md`
   负责界面文案和状态反馈规范。

这样结构、问题、UI 改造三个维度是分开的，后续继续推进时不会混乱。

## 15. 最终结论

这次前端改造的正确方向，不是“把现有 CSS 批量改成 Tailwind class”，而是：

1. 先建立设计系统。
2. 再完成结构拆分。
3. 再按区域迁移到 `shadcn-svelte + Tailwind`。
4. 再建立内容规范、上下文规则和组件治理。
5. 对通用组件标准化，对核心交互组件定制化。

只要按这个顺序推进，这次改造可以同时解决三个问题：

1. UI 一致性问题。
2. 组件复用问题。
3. `App.svelte` 和 `app.css` 的长期维护问题。
4. 文案、状态反馈和无障碍规范缺失的问题。
5. 组件没有生命周期和治理边界的问题。
